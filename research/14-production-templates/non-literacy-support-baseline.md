# Non-Literacy Support Baseline

## Purpose

This document defines the minimum support-material baseline for non-literacy curriculum packets in the repository.

It exists to prevent a two-tier system in which literacy packets receive stronger classroom support than math, science, social studies, arts, health, physical education, or technology packets.

## Required Support Set

Every governed non-literacy packet should include:

1. `teacher-guide.md`
2. `teacher-observation-checklist.md`
3. `student-scaffold.md`
4. `family-support-note.md`
5. `support-review-record.md`

## Why This Baseline Exists

Non-literacy packets still require practical classroom enactment support.

Without this layer, repository packets tend to be:

- structurally valid but harder to use live
- less consistent across subjects
- weaker for accessibility, formative assessment, and family communication

## Required Intent Of Each Artifact

### Teacher guide

Should help teachers enact the lesson well.

It should usually include:

- preparation notes
- likely misconceptions or developmental challenges
- recommended teacher moves
- observation guidance

### Teacher observation checklist

Should convert assessment or observation intent into a quick usable recording tool.

It should:

- stay brief enough for live instruction
- reflect the actual lesson objective
- guide next instructional moves rather than serve as a high-stakes evaluation

### Student scaffold

Should reduce avoidable confusion without replacing the core task.

It should:

- use short repeatable language
- support multimodal participation
- preserve the real cognitive work of the subject

### Family support note

Should extend learning at home without creating burden.

It should:

- use short plain language
- avoid assumptions about special materials or internet access
- give one or two realistic prompts or routines

### Support review record

Should document the draft-review status of the support set using the repository release review structure.

## Subject-Specific Adaptation Rules

The support baseline is shared, but the content should reflect the subject.

### Math

- emphasize representations, reasoning, and common misconceptions
- observation tools should track strategy use, not only final answers

### Science

- emphasize observation, explanation, evidence use, and visual accessibility
- family notes should avoid pseudo-scientific claims or home-lab assumptions

### Social studies

- emphasize inclusion, civic framing, and age-appropriate use of roles, place, or sources
- family notes should avoid narrow assumptions about community experience

### Arts

- emphasize creation, response, and access to varied modes of expression
- supports should not over-standardize creative output

### Health

- emphasize developmental fit, safety, and non-shaming language
- supports should be reviewed carefully for sensitivity

### Physical education

- emphasize safety, movement options, and supportive participation
- observation tools should not reward speed or ability alone

### Technology / computer science

- emphasize privacy, age-appropriate tool use, and unplugged alternatives where possible
- family notes should avoid requiring accounts, devices, or paid tools

## Definition Of Done

A non-literacy packet meets the repository support baseline when:

- all five support artifacts are present
- support artifacts reference the governed source unit and lesson
- the support artifacts remain in `Draft` until human review occurs
- the support set is included in the package review workflow

## Current Repository Implication

Going forward, new non-literacy packets should be considered incomplete until the support baseline is present.

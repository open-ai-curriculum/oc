# Computer Science Domain Comparative Analysis And Implementation Plan

## Purpose

Compare the current `computer_science` domain package with the repository's more mature domain packages at the level of:

- abstraction
- graph completeness
- package completeness
- implementation depth
- operational maturity

Then define the implementation plan required to raise the computer science domain to the same governed quality bar.

## Governing Comparison Rule

This comparison does not ask whether the domains are equal in content size.

It asks whether they are operating at the same level of:

- graph-system abstraction
- governed package completeness
- standards posture
- runtime-readiness potential
- explicit claim boundaries

## Executive Judgment

### 1. Abstraction Level

Assessment: aligned at the scaffold level

The computer science domain now defines:

- a governed domain README
- a canonical capability-graph scaffold
- a developmental spine
- an upstream build plan
- a first-node-target and wave-priority posture

At the architecture level, computer science is now inside the same repository model as mathematics, ELA/literacy, science, and learning-to-learn.

### 2. Current Implementation Depth

Assessment: computer science is materially early

Unlike mathematics and the more implemented domains, computer science does not yet have:

- a node-package layer
- domain-specific validators
- runtime slices
- artifact-attachment contracts
- graph-level learner-state contracts
- standards-native node mappings

The computer science domain is therefore structurally present but implementation-light.

### 3. Standards Posture

Assessment: intentionally deferred

Computer science does not yet have a codified external standards-support package in the repository.

Future support work likely needs explicit handling for:

- CSTA K-12 Computer Science Standards
- AP Computer Science Principles framework
- AP Computer Science A framework

However, the domain should not begin by collapsing itself into those frameworks.

### 4. Domain Ownership Boundary

Assessment: now explicit and structurally healthier

The computer science scaffold now makes explicit that:

- mathematics can serve as an upstream feeder for selected readiness
- science can consume computational-science dependencies
- AP computer science should be owned by this domain rather than absorbed into mathematics or science

## Concrete Similarities

### A. Same Repository Abstraction

Computer science now uses the same repository logic:

- domain package first
- explicit dependencies
- standards support rather than standards replacement
- claim-boundary discipline

### B. Same K-12 Continuity Rule

Computer science now preserves one continuous K-12 spine rather than treating elementary computing, middle-grade coding, and AP courses as disconnected silos.

### C. Same Upstream-First Discipline

Computer science now starts with the earliest meaningful entry points rather than beginning at AP or secondary programming only.

## Concrete Differences

### 1. Computer Science Lacks Node Packages

The more mature domains already encode:

- verification models
- failure taxonomies
- intervention playbooks
- learner-state schemas
- transition rules
- observability guidance

Computer science does not yet encode any of those at the node level.

### 2. Computer Science Lacks Graph-Level Runtime Contracts

The more mature domains already define domain-level runtime and attachment expectations.

Computer science still needs:

- graph learner-state and navigation contracts
- graph artifact-attachment posture
- initial runtime slice definitions

### 3. Computer Science Needs Standards-Structure Hardening

The domain still needs explicit repository treatment of how it will eventually support:

- CSTA standards
- AP CSP framework expectations
- AP CSA framework expectations
- mathematics dependency crosswalks
- science-domain computational attachment boundaries

### 4. Computer Science Needs Domain-Specific Failure Taxonomy Design

Computer science requires explicit differentiation among:

- interface and environment failure
- sequence and control-flow misunderstanding
- representation and data confusion
- algorithm-design weakness
- debugging and testing weakness
- systems and security misunderstanding
- impact and ethics reasoning breakdown

That layer does not yet exist.

## Implementation Plan

### Phase 1. Build The Entry Wave Node Packages

Objective:

- turn the current K-2 entry wave into the first operational computer science node layer

Deliverables:

- `D1`, `P1`, `R1`, `I1`, and `C1` node packages
- first-pass computer-science verification model patterns
- first-pass computer-science failure taxonomy patterns
- first-pass intervention playbooks and learner-state schemas

### Phase 2. Add Computer Science Graph Contracts

Objective:

- establish domain-level operational contracts before scale-up

Deliverables:

- graph learner-state and navigation contract
- graph artifact-attachment contract
- initial runtime slice schema and example slice

### Phase 3. Add Standards-Support And Cross-Domain Boundary Logic

Objective:

- support standards alignment and cross-domain dependency clarity without collapsing the domain into course identifiers

Deliverables:

- computer-science standards-support posture document
- mathematics-to-computer-science dependency map
- science-to-computational-science attachment boundary document
- claim-boundary notes for unresolved mapping questions

### Phase 4. Expand Through Elementary And Middle-Grade Waves

Objective:

- deepen the domain from entry nodes into abstraction, debugging, networks, and data reasoning

Deliverables:

- elementary bridge wave node packages
- middle-grade abstraction and systems wave node packages
- frontier-specific graph views for early and middle-grade computing coherence

### Phase 5. Deepen Advanced Secondary And AP-Facing Integration

Objective:

- connect late computer science reasoning across software, data, systems, cybersecurity, and computing impacts

Deliverables:

- advanced software and data nodes
- AP CSP and AP CSA integration nodes
- capstone and cross-domain application nodes

## Recommendation

Do not attempt to make computer science “look mature” by generating AP overlays first.

The correct path is:

1. start with the earliest entry wave
2. define strong computer-science-specific verification and failure logic
3. add graph contracts and standards/boundary logic
4. then widen the node layer

## Claim Boundary

This plan defines the implementation direction for the computer science domain. It does not mean the domain is already operationally equivalent to mathematics, science, or learning-to-learn.

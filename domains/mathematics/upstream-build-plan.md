# Mathematics Upstream Build Plan

## Purpose

Translate the zero-state to `F1` dependency spine into an executable build sequence for the repository.

## Why This Plan Exists

The current `F1 -> F5` chain is useful but incomplete. A learner cannot legitimately enter `F1` without a large body of upstream mathematical structure.

The repository therefore needs a staged build plan that expands the domain upward from the current local spine to a full developmental path.

## Build Waves

### Wave 1. Direct `F1` Predecessors

Build first:

- `R1` Part-whole reasoning with equal parts
- `R2` Unit fraction readiness
- `R3` Symbol preparation

Reason:

- these nodes are the closest true prerequisites to `F1`
- they will let the repository correct `F1` from a false root node into a real downstream node

### Wave 2. Unitizing And Measurement Foundations

Build next:

- `U1` Equal groups
- `U2` Repeated unit reasoning
- `U3` Sharing and fair partitioning
- `U4` Equal partitioning of a whole
- `M1` Measurable attributes
- `M2` Iterated units
- `M3` Unit size matters

Reason:

- these nodes create the conceptual bridge from whole-number counting to fraction structure

### Wave 3. Number Composition And Decomposition

Build next:

- `N1` Compose and decompose small numbers
- `N2` Addition and subtraction meaning
- `N3` Count on and count back
- `N4` Fluency foundations

Reason:

- `R1` depends partly on part-whole reasoning from whole-number composition, not only on partitioning

### Wave 4. Counting And Cardinality

Build next:

- `C1` Stable counting sequence
- `C2` One-to-one correspondence
- `C3` Cardinality
- `C4` Compare small quantities

Reason:

- these nodes support all later quantity, comparison, and unit reasoning

### Wave 5. Zero-State Foundations

Build next:

- `Z1` Attention to quantity
- `Z2` Match and sort objects
- `Z3` Pattern and sequence readiness

Reason:

- these are earliest-state nodes and should be modeled, but they are farther from the current fraction work than the direct predecessors are

## Operational Rule

When a new upstream wave is built:

1. add the nodes to the capability registry
2. define machine-readable node configs
3. define learner-state schemas
4. define item-bank examples
5. define failure modes and interventions
6. revise downstream node dependencies where needed

## First Revision Trigger

When `R1`, `R2`, and `R3` are complete:

- revise `F1` so its dependency set is no longer empty
- update the mathematics graph and registry to reflect the true path

## Immediate Next Target

The next best build target is the `R` family:

- `R1`
- `R2`
- `R3`
